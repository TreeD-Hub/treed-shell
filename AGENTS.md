# AGENTS.md — рабочий стандарт для `treed-shell`

## 1) Назначение

Этот файл задает постоянные правила работы агента в репозитории:
- как выполнять задачи;
- как оформлять изменения;
- как проводить аудит;
- какие проверки обязательны;
- когда можно делать commit/push.

Если запрос пользователя конфликтует с этим файлом, приоритет у явного запроса пользователя, кроме случаев небезопасных или разрушительных действий.

## 1.1) Возможности агента в этом репозитории

Агент может:
- разрабатывать UI-оболочку уровня KlipperScreen (не рескин);
- вести local-first разработку (`localhost`) с режимами `mock` и `live`;
- поддерживать целевое UI-разрешение `960x544`;
- добавлять тесты (unit/e2e/visual) и CI-конвейер;
- подготавливать runtime-артефакты для запуска на Raspberry Pi;
- поддерживать интеграционный контракт с `treed-mainshellOS`.

## 2) Язык и стиль

- Рабочий язык: русский.
- Комментарии в коде: русский, лаконично, только по делу.
- Сообщения коммитов: русский.
- Описание коммита (body): русский, кратко что/зачем/как проверено.
- Коммуникация: коротко и технически точно, без воды.

## 2.1) Формат сообщения коммита

- Заголовок коммита: короткий, предметный, в одну строку.
- Описание коммита (body): структурированное, с реальными переносами строк и списками.
- Не использовать буквальные символы `\n` в тексте коммита.
- Рекомендуемый шаблон body:

  Что:
  - пункт 1
  - пункт 2

  Зачем:
  - причина 1
  - причина 2

  Проверка:
  - команда/проверка 1
  - команда/проверка 2

## 2.2) Локальная среда агента

- Базовая локальная среда: Windows 11, shell: PowerShell.
- При локальном выполнении учитывать различия синтаксиса (`;` вместо `&&`, экранирование, кавычки, пути).
- Linux-утилиты и shell-проверки (`bash`, `sed`, `awk`, `chmod`, `find -print0`, `xargs -0`) считать локально недоступными без установленного WSL-дистрибутива.
- Если Linux-команда не исполнима локально, выполнять проверку/запуск на целевой Linux-системе (Raspberry Pi) или в CI.
- При правках `AGENTS.md` сохранять UTF-8 без потери кириллицы; перед ответом проверять `git diff AGENTS.md`.

## 2.3) Кодировка и экономия токенов

- Текстовые файлы в репозитории хранить в `UTF-8` (без смешивания с ANSI/Windows-1251).
- Не переписывать файл целиком без необходимости: вносить минимальный точечный diff.
- Для правок предпочитать `apply_patch` и короткие изменения по месту вместо генерации больших блоков текста.
- При создании временных скриптов/логов в `tmp/` не тащить их в постоянные документы и не дублировать в README/AGENTS.
- После правок с кириллицей проверять, что текст читается корректно (без «кракозябр»).

## 2.4) Антиповторы: ошибки из практики

- `PowerShell` в этой среде не принимает `&&`; использовать отдельные команды или `;`.
- Не прокидывать patch через shell как here-string для `apply_patch`: возможен сбой кодировки; применять `apply_patch` напрямую.
- В `React 19 + TypeScript` не опираться на `JSX.Element` в пропсах, если ловится `TS2503`; использовать `ReactNode`.
- Для целевого UI `960x544` не ставить media-breakpoint выше `960` (иначе включается мобильная верстка).
- При автозапуске `vite` учитывать занятый порт: dev-сервер может уйти с `4173` на `4174+`, это ломает авто-скриншоты и e2e-сценарии с жестким URL.
- Публичные `Figma Make`-роуты могут зависать на loader без авторизации/ускорения; для точного переноса требовать прямой frame-ссылки (`node-id`) или подтвержденный референс.

## 2.5) Формат комментариев в коде (зафиксированный стиль)

- Для `src/**`, `src-tauri/**`, `e2e/**`, `mocks/**`, `tools/**` использовать единый структурный формат комментариев.
- В начале сложных файлов добавлять короткий заголовок с назначением и контуром (`required`/`best-effort`/`read-only` при необходимости).
- Внутри файла размечать крупные этапы блоками вида `// Блок N: <смысловой этап>`.
- Комментарии объясняют цель и контракт блока (почему/когда), а не пересказывают синтаксис.
- При задаче «причесывание комментариев» не менять поведение кода: только комментарии/документация/пустые строки.

## 3) Ветки и Git-дисциплина

- Основная рабочая ветка: `dev`, если пользователь не указал другую.
- В `main` ничего не сливать и не пушить без прямого явного указания пользователя.
- `push` выполнять только после явного разрешения пользователя (`"коммить и пуш"`, `"пуш"` и т.п.).
- Не делать `amend`, `rebase`, `reset --hard`, удаление веток и другие потенциально опасные git-операции без явного запроса.
- Не откатывать чужие изменения без прямого указания.

## 4) Формат выполнения задач

### 4.1) Если запрос = аудит

- Режим read-only: без правок файлов.
- Выводить findings с приоритетом `P0 -> P1 -> P2`.
- Для каждого finding фиксировать:
  - факт;
  - риск;
  - действие;
  - доказательство (путь/строка).

### 4.2) Если запрос = правки

- Менять только то, что нужно для задачи, без лишнего рефакторинга.
- Сначала выполнить правки, затем проверить логику/структуру/синтаксис.
- Перед коммитом показывать краткий результат проверки.

## 5) Минимальные проверки перед commit

- Для правок в `src/**`:
  - `npm run lint`;
  - `npm run typecheck`;
  - `npm run test` (или ближайший эквивалент).
- Для правок UI/верстки/компонентов:
  - `npm run test:visual` (если настроено);
  - проверка предпросмотра `960x544` (`preview:960`).
- Для правок в `src-tauri/**`:
  - проверка сборки/рантайма `tauri:dev` или `tauri build` в доступной среде.
- Для правок shell/systemd-интеграции:
  - синтакс-проверка shell (`bash -n`) в Linux-среде или CI.
- Если часть проверок невозможно выполнить локально, явно это указывать.

## 6) Правила для UI-оболочки

- Source of truth по дизайну:
  `https://www.figma.com/make/CzDoyJ43oL0Ep8vyd93mgl/TreeD-Screen-UI-Design?t=0wAzv9BIiNp87Erv-1&preview-route=%2Flandscape`
- Foundation-токены и базовые правила зафиксированы в:
  `docs/00_FOUNDATION.md`
- Визуальные изменения UI согласовывать с этим макетом; расхождения фиксировать отдельно и явно.
- Все новые страницы и все новые элементы на страницах обязаны соответствовать `docs/00_FOUNDATION.md`.
- Нельзя добавлять новые цвета/типографику/сетку вне foundation без явного обновления `docs/00_FOUNDATION.md`.
- Для карточных UI-блоков обязательно использовать симметричные вертикальные отступы (top=bottom), а также симметричные горизонтальные отступы (left=right); наполнение центрировать по вертикали, если нет явного исключения от пользователя.
- Для типографики карточных UI-блоков использовать единые foundation-токены (`label/value/unit`) без локальных отклонений, если нет явного исключения от пользователя.
- Источник истины состояния принтера: Moonraker.
- Критичные действия (`start`, `pause`, `resume`, `cancel`, нагрев, home`) не должны выполняться «тихо» при ошибках.
- Ошибки транспорта/WS должны иметь явную индикацию в UI и контролируемый retry.
- UI обязан корректно работать в фиксированном разрешении `960x544`.
- Для долгих операций обязателен визуальный статус (loading/progress/error).
- Не ломать контур восстановления после перезапуска приложения.

## 7) Правила для интеграции с `treed-mainshellOS`

- Соблюдать контракт провайдера UI: `TREED_UI_PROVIDER=klipperscreen|treed-shell`.
- Новые переменные окружения документировать в `docs/` и интеграционной документации.
- Не добавлять silent-skip для обязательных интеграционных сценариев.
- Изменения шагов интеграции должны быть идемпотентными.
- При недоступности нового UI предусматривать управляемый fallback на `KlipperScreen`.

## 8) Документация

- Обновлять документацию только если она затронута текущей задачей или явно запрошена.
- Не переписывать docs «по пути» без необходимости.
- При правках docs сохранять соответствие фактическому коду/пайплайну.
- В каждой каталоговой директории репозитория должен быть `README.md` с кратким назначением и правилами использования.

## 8.1) Формат README (зафиксированный стиль)

- README отражает фактический код в директории, а не «планируемое» поведение.
- Базовая структура README:
  - что это за слой/каталог;
  - состав файлов или подсистем;
  - контракт/инварианты;
  - runtime/деплой-пути (если есть);
  - команды запуска/проверки (если применимо);
  - связанные документы/соседние слои.
- При удалении или переносе документа сначала обновлять ссылки на него.

## 9) Запрещено без явного запроса

- Пушить в удаленный репозиторий.
- Сливать в `main`.
- Удалять ветки (локальные/удаленные).
- Выполнять разрушительные операции с рабочим деревом или историей.
- Подключаться по SSH к Raspberry Pi без явного разрешения пользователя.
  Для разрешенного подключения использовать:
  `ssh pi@192.168.0.21` (логин `pi`, пароль `treed`).

## 10) Definition of Done

Задача считается завершенной, когда:
- выполнено ровно то, что запросил пользователь;
- логика/структура/синтаксис проверены;
- даны краткие итоги и ограничения проверки;
- commit/push сделаны только при явном разрешении пользователя.
