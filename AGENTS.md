## 4. Операционные принципы агента

### 4.1. Сначала понять, потом менять
- Не делай молчаливых предположений.
- Перед выводами проверь точку входа, композицию экрана, цепочку импортов, реально используемый слой и соседние mock/prototype-дубли.
- Если есть несколько трактовок задачи, выбирай минимально инвазивную.
- Не делай выводы по названию папки без проверки фактического использования.

### 4.2. Простое решение важнее "гибкого"
- Делай минимальное решение, достаточное для текущей задачи.
- Не добавляй новые абстракции, конфиги и расширяемость без явного запроса.
- Не предлагай смену архитектуры или стека, если задача решается в текущем контуре.

### 4.3. Правки должны быть хирургическими
- Меняй только то, что напрямую относится к задаче.
- Не рефактори соседний код без необходимости.
- Не удаляй старый код, комментарии и конструкции, если это не требуется задачей.
- Если найден побочный техдолг, фиксируй его отдельно, но не чини попутно без запроса.

### 4.4. Работай от цели, а не от расплывчатых действий
- Формулируй задачу через цель, ограничения и ожидаемый результат.
- Для многошаговых задач сначала фиксируй краткий план: шаг -> результат.
- Проверки, build, test, lint, визуальную сверку и скриншоты выполняй только по явному запросу пользователя.
- Не требовать перезапуск приложения, dev-runtime, Tauri-процесса или устройства без явного запроса пользователя либо без прямой технической необходимости задачи.
- Если проверки не запускались, явно укажи это в отчете.

### 4.5. Новые возможности Codex
- Встроенный браузер, действия по экрану и интерактивные UI-проверки использовать только там, где важен реальный рендер и поведение интерфейса: touch-flow, модальные панели, виртуальная клавиатура, переполнения, кликабельность, визуальные регрессии.
- GitHub-интеграции использовать как источник контекста для PR, review comments, check runs и CI, но выводы и правки всегда сверять с локальным кодом, diff и фактическим состоянием репозитория.
- Если данные можно подтвердить через код, типы, документацию проекта или интеграционный контракт Moonraker / `treed-mainshellOS`, сначала использовать их, а не полагаться на внешние интеграции.
- Память агента использовать только для стабильных правил работы, инвариантов проекта и долгоживущих предпочтений пользователя.
- Не сохранять в память временные баги, статусы задач, текущие ветки, промежуточные гипотезы и одноразовые решения.
- Автоматизации и отложенные напоминания использовать только по явной договоренности с пользователем: что делать, когда запускать и когда останавливать.
- Параллельных агентов использовать только если задача делится на независимые подзадачи с несмешивающимися зонами ответственности.

### 4.6. Работа с инструментами и shell
- Команды писать под текущий shell проекта: `PowerShell`. Не использовать bash-специфичные конструкции вроде `&&`, `||`, `<(...)>` и похожие сокращения, которые не поддерживаются текущей средой.
- Не использовать `rg` как основной инструмент поиска. Для поиска по репозиторию сначала использовать `git grep -n`.
- При работе с файлами, где есть кириллица, не полагаться на текст из консоли как на точный источник для патча. Перед изменениями читать файл через `Get-Content -Encoding UTF8`.
- Перед поиском сначала сузить область: экран -> feature -> директория -> тип файлов. Не делать широкие repo-wide поиски без необходимости.
- Для поиска по коду предпочитать несколько простых и устойчивых команд вместо одного хрупкого one-liner с длинным regex и сложным экранированием.
- По умолчанию не запускать команды с потенциально длинным или шумным выводом без ограничения объема: использовать `Select-Object -First`, точечные диапазоны строк, `-TotalCount`, узкие `Select-String`, адресный `git diff -- <path>`.
- Перед любым чтением файла сначала выбрать минимальный фрагмент. Не печатать целиком большие файлы, если для решения нужен только конкретный блок.
- Если команда уже один раз уперлась в timeout или дала слишком большой вывод, не повторять ее в том же виде. Сразу сузить диапазон, путь, шаблон или разбить просмотр на короткие куски.
- Не угадывать путь к файлу. Если точный путь не подтвержден через импорты, список файлов или точку входа, сначала найди фактическое расположение.
- Не перечитывать одни и те же опорные файлы и не дергать `git status` / `git diff` повторно без новой причины.
- Если вопрос не блокирует работу и есть безопасный дефолт, фиксируй допущение и продолжай, а не раздувай задачу лишними уточнениями.
- Перед `build` / `test` / browser / Playwright-проверкой сначала убедись, что это явно запрошено пользователем и что среда к этому готова. Если базовые предпосылки не выполнены, остановись и кратко зафиксируй ограничение вместо серии неуспешных попыток.

## 6. Технологический контекст проекта

- React 19
- TypeScript
- Vite
- Tauri 2
- local-first разработка с режимами `mock` и `live`
- источник истины состояния принтера: Moonraker

Ключевые точки проекта:
- точка входа UI: `src/main.tsx`
- основная экранная композиция: `src/App.tsx`
- команды принтера: `src/core/commands/**`
- транспорт и клиенты Moonraker: `src/core/transport/**`
- снимок и доступ к состоянию: `src/core/store/**`
- UI primitives и переиспользуемые виджеты: `src/ui/**`
- экранные конфиги и вспомогательная логика: `src/dashboard/**`
- runtime-оболочка и конфиг приложения: `src-tauri/**`
- foundation и дизайн-правила: `docs/00_FOUNDATION.md`
- e2e-контур: `e2e/**`
- mocks и локальные заглушки: `mocks/**`

Важно:
- приложение сейчас не опирается на `react-router`;
- переключение экранов и состояний нужно сверять по фактической логике `src/App.tsx`;
- `mock` и `live` нельзя считать взаимозаменяемыми без проверки контракта и точки подключения.

## 7. Инварианты проекта

### 7.1. Архитектура оболочки
Нельзя ломать:
- local-first сценарий разработки;
- Tauri-контур и конфиги `src-tauri/**`;
- сохранность пользовательского состояния приложения;
- базовую экранную композицию, если задача не требует ее менять явно.

### 7.2. Moonraker / transport / команды
Для задач, связанных с backend, состоянием принтера, payload/response и управляющими командами, первым шагом обязательно сверяй:
- типы и реализацию в `src/core/commands/**`;
- транспорт и обработку ошибок в `src/core/transport/**`;
- фактическое потребление состояния в `src/core/store/**`.

Нельзя ломать:
- источник истины состояния принтера через Moonraker;
- семантику критичных действий `start`, `pause`, `resume`, `cancel`, нагрев, `home`, калибровки;
- явную индикацию ошибок транспорта и контролируемый retry;
- переключение `mock/live`, если это не является прямой задачей.

Без явной задачи нельзя:
- менять контракт Moonraker-клиента;
- добавлять silent-fail для критичных действий;
- менять семантику обработки потери соединения;
- менять формат ключевых данных, на которые опирается UI принтера.

### 7.3. Интеграция с `treed-mainshellOS`
Контекст:
- этот репозиторий отвечает за UI-клиент `treed-shell`, а не за железный/system runtime;
- железо, системный запуск, fallback и текущий стоковый UI находятся в зоне ответственности [`TreeD-Hub/treed-mainshellOS`](https://github.com/TreeD-Hub/treed-mainshellOS);
- любые изменения, которые делают `treed-shell` runtime-провайдером вместо стокового UI, требуют явного интеграционного контракта.

Нельзя ломать:
- контракт провайдера UI: `TREED_UI_PROVIDER=klipperscreen|treed-shell`;
- обязательные интеграционные сценарии старта и fallback;
- идемпотентность интеграционных шагов.

Без явной задачи нельзя:
- менять поведение fallback на `KlipperScreen`;
- вводить новые обязательные env-переменные без обновления документации;
- убирать или обходить обязательные шаги интеграции.

### 7.4. UI и дизайн-система
- Основной UI-слой: `src/ui/**`.
- Базовые визуальные правила и токены: `docs/00_FOUNDATION.md`.
- Глобальные стили приложения: `src/index.css`, `src/App.css`.
- Повторяемые кнопки, карточки, метрики, переключатели и control widgets выносить в `src/ui`, а не копировать локально в экран.
- Не тащить в проект чужой дизайн-язык без прямого запроса, кроме явно зафиксированного ниже направления.

Стиль всего интерфейса:
- интерфейс должен быть в направлении `Nothing-inspired`;
- ориентироваться на принципы из Refero `DESIGN.md`, а не на слепое копирование чужого сайта;
- базовые референсы:
  - `https://styles.refero.design/style/4acc13a0-c553-40d7-b78d-a9b6a4e94486`
  - `https://styles.refero.design/style/4433dfe7-315a-4459-bfd7-f59ccdc09bad`
  - `https://styles.refero.design/style/e4a7b5f3-f393-4f6d-b4a5-ecf874024bed`
  - дополнительные референсы по задаче: `https://styles.refero.design/style/6d0db1c7-5500-40cd-b2f3-d3d9abbd3a2f` и пользовательские ссылки из задачи
- брать из референсов:
  - высокий контраст и темную/монохромную базу;
  - точную индустриальную типографику;
  - плоские поверхности, тонкие границы, минимум декоративного шума;
  - сдержанный акцентный цвет только для статусов, CTA и критичных действий;
  - ощущение "device UI", а не маркетингового лендинга.
- не переносить в интерфейс:
  - тяжелые градиенты;
  - случайные яркие цвета;
  - мелкие hover-only паттерны;
  - декоративные тени, не несущие состояния.

При обсуждении UI-задач отдельно учитывать:
- адаптив;
- переполнение;
- симметрию;
- кликабельность;
- базовую доступность;
- читаемость с близкой дистанции на экране принтера.

### 7.5. Touch-first и 5-дюймовый экран
- Интерфейс обязательно должен быть адаптирован под сенсорный 5-дюймовый экран.
- Базовый рабочий canvas: `960x544`.
- Любой основной сценарий должен быть доступен без hover и без точных мышиных движений.
- Интерактивные зоны не должны быть мелкими или плотно слипшимися. Для основных touch-элементов ориентир - не меньше `48x48px`.
- Нельзя строить ключевые сценарии на длинных таблицах, мелких чекбоксах, узких полосах прокрутки и скрытых hit-area.
- Верхние и нижние панели, модалки, клавиатура, системные кнопки и критичные действия должны помещаться в пределах 5-дюймового touch-экрана без горизонтального скролла.
- Приоритет у краткого текста, крупных статусов, явных touch-target и предсказуемой иерархии.

## 8. Режимы работы

### 8.1. Если запрос = навигация по репо
Нужно ответить в таком порядке:
1. Наиболее вероятное место.
2. Что проверить рядом.
3. Какие mock/prototype-дубли не перепутать.

### 8.2. Если запрос = постановка задачи
Нужно сформулировать:
1. Краткий вывод.
2. Куда это относится в проекте.
3. Как лучше поставить задачу.
4. Что не сломать.
5. Ожидаемый результат.
6. Какие проверки уместны после выполнения.

### 8.3. Если запрос = аудит
Режим read-only.

Формат findings:
- `P0` / `P1` / `P2`
- факт
- риск
- действие
- доказательство: путь и, если возможно, участок кода

### 8.4. Если запрос = правки
- Менять только то, что нужно для задачи.
- Не делать лишний рефакторинг.
- Не менять бизнес-логику команд принтера, интеграционный контракт, transport-flow и Tauri-контур без прямого запроса.
- Перед редактированием коротко зафиксировать, какие именно файлы и блоки будут изменены, и дальше работать только в этом контуре.
- Предпочитать один цельный пакет правок на файл вместо серии мелких patch-итераций.
- Не переписывать файл целиком, если задача локальная. Полная замена файла допустима только если пользователь прямо этого хочет или локальная правка технически невозможна без риска.
- Если патч к одному и тому же файлу дважды подряд не применился или привел к мусорному промежуточному состоянию, остановиться, перечитать минимальный нужный фрагмент и только потом делать следующую попытку.
- После неудачной правки нельзя "допереписывать" файл вслепую серией новых патчей. Сначала нужно подтвердить текущее состояние файла и сократить план до минимального исправления.
- После правок кратко описать, что именно изменено.
- Отдельно указать, какие проверки были запрошены и выполнены.
- Если проверки не запускались, явно указать это.

## 10. Правила формулирования задач

Когда формируешь задачу для разработки, по умолчанию включай:

### 10.1. Контекст
Кратко укажи:
- что это за проект;
- какой стек используется;
- где точка входа и где основная экранная композиция;
- какие слои актуальны;
- где проходят команды, transport и состояние;
- что в проекте есть `mock/live` и интеграция с `treed-mainshellOS`.

### 10.2. Цель
Опиши проверяемый результат.
Не общими словами, а как конкретный итог.

### 10.3. Ограничения
Всегда фиксируй:
- менять только то, что нужно для задачи;
- сначала проверить реальные импорты;
- менять только реально используемый слой;
- не делать массовый рефакторинг;
- не менять бизнес-логику команд принтера, API Moonraker, Tauri-интеграцию и fallback без прямого запроса.

### 10.4. Ожидаемый результат
Опиши:
- где именно должны быть изменения;
- что пользователь должен увидеть или получить;
- что должно остаться без изменений;
- как поведение должно выглядеть на 5-дюймовом touch-экране.

### 10.5. Проверки
В конце задачи указывай:
- что проверить вручную;
- какие сценарии не должны сломаться;
- какие регрессии особенно вероятны;
- что проверить отдельно на canvas `960x544`.

## 11. Специальные области риска

### 11.1. Критичные сценарии принтера
Для задач, связанных с `start`, `pause`, `resume`, `cancel`, нагревом, `home`, парковкой, калибровками и сетевыми состояниями, отдельно проверять:
- корректность вызова команды;
- поведение при ошибке транспорта;
- состояние UI до, во время и после действия;
- защиту от "тихого" сбоя;
- читаемость и кликабельность на touch-экране.

### 11.2. Крупные панели и виртуальная клавиатура
Для настроек, виртуальной клавиатуры, длинных модалок и tune-панелей отдельно проверять:
- помещаются ли элементы в `960x544`;
- не перекрывает ли клавиатура критичные поля и CTA;
- нет ли переполнений и скрытых элементов;
- можно ли завершить сценарий без мыши и hover.

### 11.3. Mock / live / e2e
- Основной e2e-контур: `e2e/**`.
- Заглушки и локальные данные: `mocks/**`.
- Не смешивать `mock`-предположения с `live`-поведением без явной задачи.
- Перед выводами по e2e и интеграции учитывать, к какому режиму относится конкретный сценарий.

## 14. Git-дисциплина

- Основная рабочая ветка по умолчанию: `dev`, если пользователь не указал другую.
- В `main` ничего не сливать и не пушить без прямого явного указания пользователя.
- `push` выполнять только после явного разрешения пользователя.
- Не делать `amend`, `rebase`, `reset --hard`, удаление веток и другие потенциально опасные git-операции без прямого запроса.
- Не откатывать чужие изменения без прямого указания.

### 14.1. Оформление commit
- Коммит должен быть атомарным: одна логическая задача = один коммит.
- В коммит не включать нерелевантные изменения и побочные правки "заодно".
- Язык commit message по умолчанию: русский, если пользователь не попросил иначе.
- Заголовок commit: короткий, предметный, в одну строку.
- Body commit: структурированный, с реальными переносами строк.
- Рекомендуемый шаблон body:
  - `Что:` список ключевых изменений;
  - `Зачем:` причина/контекст;
  - `Проверка:` что реально запускалось, либо явно `не запускалось`.

### 14.2. Перед рекомендацией commit
- Перед рекомендацией commit проверить, что staged/измененные файлы относятся только к текущей задаче.
- В отчете перед commit кратко фиксировать, какие проверки выполнены, а какие не запускались.

### 14.3. GitHub / PR / CI
- GitHub-интеграции использовать для чтения PR, review comments, check runs и CI-логов, когда задача прямо связана с PR, комментариями ревью или падением пайплайна.
- Не создавать PR, не публиковать draft PR, не отвечать комментариями и не делать другие write-действия в GitHub без явного запроса пользователя.
- При анализе замечаний ревью или CI сначала подтвердить, что они относятся к актуальному коду и текущей ветке, а не к устаревшему diff.

## 17. Правила адаптации интерфейса 3D-принтера

### 17.1. Базовый экранный стандарт
- Целевой интерфейс - это UI 3D-принтера, а не desktop-first веб-панель.
- Основной сценарий использования - сенсорный 5-дюймовый экран.
- Базовое разрешение проектирования и визуальной проверки: `960x544`.
- Любой новый экран, модалка или control-панель должны проектироваться под этот размер в первую очередь.

### 17.2. Touch-first поведение
- Приоритет у крупных touch-target, коротких цепочек действий и явных состояний.
- Долгие операции обязаны иметь заметный статус: `loading`, `progress`, `success`, `error`.
- Критичные действия должны иметь явное подтверждение или настолько читаемое состояние, чтобы пользователь не ошибся касанием.
- Нельзя рассчитывать на hover, tooltip-on-hover или тонкие микрозоны попадания.

### 17.3. Плотность и компоновка
- Не допускать горизонтального скролла на ключевых экранах.
- Не прятать важные элементы за невидимыми overflow-областями.
- Для длинных настроек, списков файлов и tune-панелей сначала думать о chunking, пагинации, вкладках или модалках, а не о сжатии текста до нечитаемого состояния.
- Верхняя и нижняя системная навигация не должны конфликтовать с основным контентом.

### 17.4. Визуальный характер
- Весь интерфейс держать в направлении `Nothing-inspired device UI`.
- Визуальный язык: монохромная база, высокий контраст, точная сетка, индустриальная типографика, аккуратные границы, минимальный декоративный шум.
- Акцентные цвета использовать дозированно: статусы, прогресс, CTA, критичные действия.
- По умолчанию избегать "маркетинговых" градиентов, стеклянных эффектов и случайной иллюстративности.
- Любой новый визуальный паттерн должен быть совместим с `docs/00_FOUNDATION.md` и слоем `src/ui/**`.

### 17.5. Что проверять в UI-задачах
- Читаемость с близкой дистанции.
- Размер touch-target.
- Переполнение текстов и карточек.
- Помещение модалок и клавиатуры в `960x544`.
- Отсутствие случайных состояний, которые можно активировать только мышью.
